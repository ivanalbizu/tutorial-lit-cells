Paso 5 — Eventos y comunicación

CustomEvent · dispatchEvent · padre ↔ hijo

El patrón de comunicación

Flujo de datos predecible

⬇ Padre → Hijo: propiedades (.prop=${valor})

⬆ Hijo → Padre: eventos (dispatchEvent)

El padre escucha los eventos, actualiza su estado y le pasa los nuevos valores al hijo como propiedades

Anatomía de un CustomEvent

Las tres propiedades esenciales


this.dispatchEvent(new CustomEvent('rating-changed', {
  detail:   { rating: 4 },
  bubbles:  true,
  composed: true,
}));
  • detail — el payload del evento; puede ser cualquier objeto
  • bubbles: true — el evento sube por el árbol DOM hasta el documento
  • composed: true — el evento cruza el Shadow DOM

Sin composed: true, el evento queda atrapado dentro de la shadow root y el componente padre nunca lo recibe.

my-rating.ts

Componente hijo — propiedades de entrada


@customElement('my-rating')
export class MyRating extends LitElement {

  @property({ type: Number })
  value = 0;

  @property({ type: Number })
  max = 5;

}

Ambas propiedades vienen del exterior — por eso son @property

my-rating.ts

Emitir el evento al seleccionar una estrella


  private _select(rating: number) {
    this.dispatchEvent(new CustomEvent('rating-changed', {
      detail: { rating },
      bubbles: true,
      composed: true,
    }));
  }

Convención: nombres de eventos en kebab-case — rating-changed, item-selected...

El padre escuchará @rating-changed en su template, igual que un @click nativo

my-rating.ts

Render — botones de estrellas


  render() {
    return html`
      ${Array.from({ length: this.max }, (_, i) => html`
        <button
          @click=${() => this._select(i + 1)}
          aria-label="Valorar con ${i + 1} estrellas">
          ${i < this.value ? '★' : '☆'}
        </button>
      `)}
    `;
  }

aria-label hace el componente accesible para lectores de pantalla

my-review.ts

Componente padre — gestiona el estado


@customElement('my-review')
export class MyReview extends LitElement {

  @state()
  private _reviews: Review[] = [];

  @state()
  private _currentRating = 0;

  @state()
  private _inputText = '';

}

_currentRating almacena la valoración que el usuario está eligiendo antes de enviar el formulario

my-review.ts

Handler que escucha el evento del hijo


private _onRatingChanged(e: CustomEvent<{ rating: number }>) {
  this._currentRating = e.detail.rating;
}

private _submit() {
  if (!this._currentRating) return;
  this._reviews = [
    ...this._reviews,
    { id: Date.now(), rating: this._currentRating, text: this._inputText },
  ];
  this._currentRating = 0;
  this._inputText = '';
}

El CustomEvent se tipa con la forma del detail: CustomEvent<{ rating: number }>
TypeScript garantiza que accedemos correctamente a e.detail.rating.

my-review.ts

Render — el binding @rating-changed


  render() {
    return html`
      <my-rating
        .value=${this._currentRating}
        @rating-changed=${this._onRatingChanged}>
      </my-rating>
      <textarea
        .value=${this._inputText}
        @input=${(e: InputEvent) =>
          this._inputText = (e.target as HTMLTextAreaElement).value}>
      </textarea>
      <button ?disabled=${!this._currentRating} @click=${this._submit}>
        Enviar reseña
      </button>
      ${this._reviews.map(r => html`
        <div class="review-item">
          <my-rating .value=${r.rating}></my-rating>
          <p>${r.text || html`<em>Sin comentario</em>`}</p>
        </div>
      `)}
    `;
  }

Simetría perfecta:
hijo emite rating-changed → padre escucha @rating-changed

composed: true — por qué importa

Demostración: qué pasa sin él


this.dispatchEvent(new CustomEvent('rating-changed', {
  detail: { rating },
  bubbles: true,
  // composed: true  ← comentado
}));

Sin composed: true: el evento queda atrapado dentro del Shadow DOM de my-rating.
El padre nunca lo recibe. La valoración no cambia al hacer clic.

Siempre incluye composed: true en eventos de componentes que viven dentro de un Shadow DOM

Listener manual — sin @Listen

Para casos fuera del template: connectedCallback


connectedCallback() {
  super.connectedCallback();
  this.addEventListener('rating-changed', this._onRatingChanged);
}

disconnectedCallback() {
  super.disconnectedCallback();
  this.removeEventListener('rating-changed', this._onRatingChanged);
}

No existe decorador @Listen en Lit.
Listener manual para casos especiales: escuchar en el propio componente raíz, en window o en document.

El removeEventListener en disconnectedCallback es obligatorio para evitar fugas de memoria

Resumen

  • Datos hacia abajo con propiedades — eventos hacia arriba
  • dispatchEvent(new CustomEvent(...)) emite desde el hijo
  • detail — el payload; puede ser cualquier objeto
  • bubbles: true — el evento sube por el DOM
  • composed: true — cruza el Shadow DOM (sin él, el padre no lo recibe)
  • @nombre-evento=${handler} — igual que cualquier evento nativo
  • No hay @Listen — usar addEventListener en connectedCallback y limpiar en disconnectedCallback

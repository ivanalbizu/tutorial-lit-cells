Acceso declarativo al DOM interno del componente

El problema que resuelven

Sin decoradores: acceso manual verboso

@query, @queryAll y @queryAsync son atajos declarativos con tipos correctos de TypeScript.

@query — elemento único

Getter con caché para el primer elemento que coincide


import { LitElement, html, css } from 'lit';
import {
  customElement, query, queryAll, queryAsync, state
} from 'lit/decorators.js';

@customElement('my-form-demo')
export class MyFormDemo extends LitElement {

  // Getter cacheado — llama querySelector solo la primera vez
  @query('#name-input')
  private nameInput!: HTMLInputElement;

  @query('#email-input')
  private emailInput!: HTMLInputElement;

  @query('button[type="submit"]')
  private submitBtn!: HTMLButtonElement;

  // ... métodos y render en slides siguientes
}

El ! en el tipo (nameInput!: HTMLInputElement) le dice a TypeScript que garantizamos que el elemento existe en el DOM.

@query — caché y advertencia


  // Caché activado por defecto
  @query('#name-input')
  private nameInput!: HTMLInputElement;

  // Desactivar caché: busca en el DOM en cada acceso
  @query('#name-input', true)
  private nameInputNoCache!: HTMLInputElement;

No accedas a propiedades @query en el constructor ni en connectedCallback antes del primer render. El Shadow DOM aún no existe.

cache: false solo tiene sentido para elementos condicionales con acceso síncrono.
Para ese caso, @queryAsync es la solución idiomática — lo vemos en el siguiente slide.

@queryAll — múltiples elementos

Equivalente a querySelectorAll — devuelve NodeList


  // NodeList con todos los inputs del formulario
  @queryAll('input')
  private allInputs!: NodeListOf<HTMLInputElement>;

  private _validate(): boolean {
    let valid = true;

    // Iterar sobre todos los inputs con allInputs
    this.allInputs.forEach(input => {
      if (!input.value.trim()) {
        input.classList.add('invalid');
        valid = false;
      } else {
        input.classList.remove('invalid');
      }
    });

    return valid;
  }

@queryAll no se cachea — devuelve una NodeList viva en cada acceso

@queryAsync — elementos condicionales

Devuelve una Promise que resuelve tras el siguiente render


  @state() private _submitted = false;

  // El elemento #success-message solo existe cuando _submitted === true
  @queryAsync('#success-message')
  private successMessage!: Promise<HTMLElement | null>;

Si usáramos @query aquí, el elemento aún no estaría en el DOM: Lit renderiza de forma asíncrona.

@queryAsync resuelve después del siguiente ciclo de renderizado. Siempre úsalo con await.

@queryAsync — uso con await


  private async _handleSubmit(e: Event) {
    e.preventDefault();
    if (!this._validate()) return;

    this._submitted = true; // dispara un nuevo render

    // Esperamos a que el render termine y el elemento aparezca
    const msg = await this.successMessage;
    msg?.scrollIntoView({ behavior: 'smooth' });
    msg?.focus();
  }

  private _handleReset() {
    this._submitted = false;
    // updateComplete: Promise que resuelve cuando el DOM está listo
    this.updateComplete.then(() => {
      this.nameInput.focus();
    });
  }

updateComplete es la alternativa síncrona a @queryAsync: úsala cuando ya tienes el elemento con @query pero necesitas esperar al render.

Flujo completo

Por qué cada decorador en su lugar

@query('#name-input') — acceso directo y cacheado. Se usa en _handleReset para enfocar tras el render.

@queryAll('input') — itera todos los campos para validar en lote.

@queryAsync('#success-message') — el elemento solo existe cuando _submitted es true. Necesita la Promise para esperar al render.

Resumen

@query('#selector')
  → primer elemento — cacheado por defecto
  → { cache: false } para desactivar caché

@queryAll('.selector')
  → NodeList — todos los elementos que coinciden

@queryAsync('#selector')
  → Promise — resuelve tras el siguiente render
  → necesario para elementos condicionales

updateComplete
  → Promise de Lit — DOM listo tras el render actual
